## 1 — WORKSPACE PROVISIONING & ISOLATION (Controls WOS-001 through WOS-012)

| Control ID | TSC Reference | Description | Type | Implementation Details | Testing Procedure | Frequency | Evidence | Owner |
|---|---|---|---|---|---|---|---|---|
| WOS-001 | CC6.1, CC6.3 | **Tenant Isolation at Provisioning** — Each customer workspace is provisioned as a completely separate Google Workspace instance with a unique organization unit (OU) hierarchy, dedicated super-admin account, and logically isolated domain. No shared organizational resources exist across customer boundaries. | Preventive | 1. Provisioning engine creates a new Google Workspace tenant via the Google Reseller API. 2. A unique primary domain (customer-slug.workspaceos.io) is registered. 3. Dedicated OU tree is auto-generated per the WorkspaceOS Baseline Template. 4. Domain verification DNS records are set in the customer's isolated DNS zone. 5. Provisioning event is logged in the Tenant Lifecycle Ledger (immutable append-only log). | 1. Select 25 randomly sampled tenants provisioned during the audit period. 2. Verify each tenant has a unique Customer ID, primary domain, and OU path. 3. Confirm no OU or domain overlaps with any other customer. 4. Inspect the Tenant Lifecycle Ledger for provisioning event integrity. 5. Attempt cross-tenant resource resolution and confirm denial. | Each provisioning event; quarterly sampling audit | Tenant Lifecycle Ledger entries; Google Admin API export showing distinct org units; provisioning job logs; DNS zone file snapshots | Platform Engineering Lead |
| WOS-002 | CC6.1, CC6.6 | **Cross-Tenant Network Segmentation** — Network-level controls ensure that no lateral traffic can traverse between customer workspace environments. Each tenant's management plane operates within an isolated VPC or project boundary in the WorkspaceOS control plane infrastructure. | Preventive | 1. Each tenant management sidecar runs in a dedicated GCP project with VPC Service Controls perimeter. 2. Firewall rules default-deny all ingress/egress between tenant projects. 3. Shared services (logging, monitoring) are accessed only through Private Service Connect endpoints with IAM-scoped access. 4. VPC flow logs are enabled on all subnets. | 1. Review GCP Organization hierarchy and confirm one project per tenant. 2. Inspect VPC Service Controls perimeters for correct scoping. 3. Attempt network connectivity tests between two tenant projects (ping, curl, port scan) and confirm failure. 4. Review VPC flow logs for any unauthorized cross-project traffic. | Continuous (flow logs); quarterly penetration test | VPC Service Controls configuration export; firewall rule sets; VPC flow log samples; penetration test report | Cloud Infrastructure Manager |
| WOS-003 | CC6.1, CC6.3, C1.1 | **Data Store Isolation** — All customer configuration data, audit logs, and metadata are stored in tenant-scoped data partitions. No multi-tenant shared database tables exist without cryptographic tenant-key separation. | Preventive | 1. Each tenant's configuration state is stored in a dedicated Cloud Firestore database (or namespace) keyed by Tenant ID. 2. Encryption at rest uses per-tenant Cloud KMS keys (Customer-Managed Encryption Keys where contracted). 3. All database queries are executed through a Data Access Layer (DAL) that programmatically injects Tenant ID predicates into every query. 4. Row-level security policies enforced at the database layer as a secondary control. | 1. Examine DAL source code for mandatory Tenant ID injection (code review). 2. Execute SQL/NoSQL queries without Tenant ID and confirm rejection. 3. Verify per-tenant KMS key existence for 25 sampled tenants. 4. Attempt to read Tenant A data while authenticated as Tenant B and confirm access denial. | Code review each release; quarterly isolation penetration test | Source code repository commit history; KMS key inventory; penetration test report; DAL unit test results | Platform Engineering Lead |
| WOS-004 | CC6.2, CC6.3 | **Unique Super-Admin Credential Generation** — During provisioning, each customer workspace receives a unique super-admin account with a cryptographically random password stored in a dedicated secrets vault partition. WorkspaceOS operators never share admin credentials across tenants. | Preventive | 1. Provisioning pipeline generates a 24-character password (uppercase, lowercase, digits, symbols) using a CSPRNG. 2. Credential is stored in HashiCorp Vault under path `secret/tenants/{tenant-id}/super-admin`. 3. Vault policy restricts access to the provisioning service account and break-glass operators only. 4. Password is rotated every 90 days automatically. 5. Customer is provided secure onboarding portal link (one-time use) to set their own admin credentials. | 1. Review Vault policies for tenant credential paths. 2. Verify password complexity by inspecting generation function source code. 3. Confirm one-time onboarding link expiry for 10 sampled tenants. 4. Attempt to access Tenant A credentials with Tenant B's service account and confirm denial. 5. Verify 90-day rotation schedule via Vault audit logs. | Each provisioning event; quarterly Vault audit | Vault audit logs; provisioning pipeline code; onboarding portal access logs; password policy configuration | Identity & Access Management Lead |
| WOS-005 | CC6.1, CC6.7, A1.1 | **Baseline Configuration Template Enforcement** — Every new workspace is provisioned with a hardened Google Workspace configuration baseline that includes disabled legacy protocols, enforced 2-Step Verification, restricted external sharing defaults, and DLP policies. | Preventive | 1. Configuration-as-Code baseline defined in Terraform/Pulumi referencing the WorkspaceOS Hardening Standard (v2.4). 2. Baseline applied atomically during provisioning via Google Workspace Admin SDK and GAM (Google Workspace command-line tool). 3. Post-provisioning validation job runs 46 checks against the baseline. 4. Any deviation blocks tenant activation and alerts the Provisioning Team. | 1. Obtain the baseline template and compare against the WorkspaceOS Hardening Standard. 2. Select 25 tenants and run the 46-check validation job; confirm 100% pass rate. 3. Attempt to activate a tenant with a deliberately failed check and confirm activation is blocked. 4. Review alert history for any baseline drift notifications. | Each provisioning; continuous drift detection (hourly) | Baseline template (IaC repository); validation job output reports; drift detection alert logs; Google Workspace Admin audit logs | Security Engineering Manager |
| WOS-006 | CC6.6, CC6.7 | **Provisioning Pipeline Integrity** — The automated provisioning pipeline is protected against tampering through signed commits, immutable container images, and pipeline-as-code stored in a version-controlled repository with mandatory code review. | Preventive | 1. All provisioning code resides in a protected branch requiring 2 approvals from the Platform Engineering team. 2. Commits are GPG-signed. 3. CI/CD pipeline (GitHub Actions / Cloud Build) builds container images with content-addressable digests. 4. Binary Authorization policy enforces that only images signed by the CI service account can be deployed. 5. Pipeline secrets injected at runtime from Vault; never stored in code. | 1. Review branch protection rules and confirm 2-approval requirement. 2. Verify GPG signature on 25 recent commits. 3. Inspect Binary Authorization policy for attestation requirements. 4. Attempt to deploy an unsigned image and confirm rejection. 5. Scan repository for embedded secrets using TruffleHog/GitLeaks. | Each deployment; quarterly pipeline integrity audit | Branch protection configuration; GPG key registry; Binary Authorization policy; CI/CD pipeline run logs; secret scan reports | DevOps Lead |
| WOS-007 | CC6.3, C1.2 | **Tenant Deprovisioning & Data Purge** — When a customer offboards, a controlled deprovisioning process ensures complete data destruction within contracted timeframes, with cryptographic verification of deletion. | Corrective | 1. Deprovisioning initiated via support ticket + customer written confirmation. 2. Automated pipeline suspends workspace, exports final data archive to customer-accessible bucket (30-day retention). 3. After retention window, pipeline deletes: Google Workspace tenant, GCP project, Vault secrets, database namespace, KMS keys (scheduled destruction with 24-hour delay). 4. Deletion certificate generated with SHA-256 hashes of destroyed resources. 5. Tenant ID added to the revocation list preventing re-use. | 1. Review 10 deprovisioning events from the audit period. 2. Confirm customer authorization documentation exists. 3. Verify data archive delivery and retention window compliance. 4. Confirm all resource types were destroyed (query each system for tenant ID). 5. Validate deletion certificate hash integrity. 6. Attempt to access any resource using the deprovisioned Tenant ID. | Each deprovisioning event; annual full lifecycle test | Deprovisioning ticket records; customer authorization emails; deletion certificates; resource query results; KMS key destruction logs | Customer Success Lead & Platform Engineering Lead |
| WOS-008 | CC6.1, PI1.3 | **Provisioning Idempotency & Rollback** — The provisioning pipeline is designed to be idempotent. If a provisioning job fails mid-execution, automated rollback removes all partially created resources to prevent orphaned tenant fragments. | Corrective | 1. Provisioning state machine tracks each step (domain registration → OU creation → admin account → baseline config → validation). 2. On failure, compensating transactions execute in reverse order. 3. Orphan detection job runs every 6 hours, scanning for resources without a matching active Tenant ID. 4. Orphaned resources are quarantined and reported for manual review within 24 hours. | 1. Deliberately inject a failure at step 3 of provisioning and observe rollback behavior. 2. Verify no orphaned resources remain after rollback. 3. Review orphan detection job logs for the audit period. 4. Confirm quarantine-and-review process for any detected orphans. | Each provisioning event (automatic); orphan detection every 6 hours | State machine execution logs; rollback transaction logs; orphan detection reports; quarantine review tickets | Platform Engineering Lead |
| WOS-009 | CC6.3, CC8.1 | **Tenant Naming & Identifier Uniqueness** — Every tenant is assigned a globally unique, immutable Tenant UUID that serves as the primary key across all WorkspaceOS subsystems. Human-readable slugs are unique within a separate namespace and mapped 1:1 to UUIDs. | Preventive | 1. UUID v4 generated at provisioning inception. 2. Uniqueness enforced via database unique constraint and pre-insertion check. 3. Slug uniqueness enforced at the API gateway with conflict detection (HTTP 409). 4. Neither UUID nor slug can be modified after creation. 5. All internal APIs require Tenant UUID; slugs are resolved only at the edge. | 1. Attempt to provision two tenants with the same slug and confirm rejection. 2. Attempt to modify a Tenant UUID via direct database access and confirm immutability constraint. 3. Verify UUID presence in 25 sampled cross-system records (Vault, database, GCP project labels, DNS). | Each provisioning event; annual identifier integrity audit | Database schema showing unique constraints; API gateway conflict logs; cross-system identifier audit report | Platform Engineering Lead |
| WOS-010 | CC6.1, A1.2 | **Provisioning Rate Limiting & Abuse Prevention** — Provisioning API endpoints enforce rate limits to prevent abuse, resource exhaustion, or denial-of-service attacks against the provisioning infrastructure. | Preventive | 1. API gateway enforces per-IP and per-authenticated-principal rate limits (5 provisioning requests/hour, 20/day). 2. CAPTCHA challenge required for unauthenticated provisioning trial requests. 3. Provisioning queue has max depth of 50; overflow requests receive HTTP 429 with Retry-After header. 4. Anomaly detection alerts on burst provisioning patterns. | 1. Submit provisioning requests exceeding rate limits and confirm HTTP 429 responses. 2. Review rate limit configuration in API gateway. 3. Verify queue depth limits and overflow behavior. 4. Review anomaly detection alert history. | Continuous enforcement; quarterly limit review | API gateway configuration; rate limit test results; anomaly detection alert logs; queue depth monitoring dashboard | API Security Lead |
| WOS-011 | CC6.1, CC7.1 | **Provisioning Approval Workflow for Enterprise Tier** — Enterprise-tier customer provisioning requires manual approval by a WorkspaceOS Solutions Architect after identity verification, contract review, and capacity planning assessment. | Preventive | 1. Enterprise provisioning requests routed to approval queue in ServiceNow. 2. Solutions Architect verifies: customer identity (domain ownership), signed MSA, capacity requirements. 3. Two-person approval required (Solutions Architect + Finance). 4. Approved requests trigger automated provisioning with audit trail linking to approval ticket. 5. SLA: approval within 4 business hours. | 1. Review 15 enterprise provisioning events and confirm dual-approval documentation. 2. Verify identity verification artifacts (domain ownership proof, MSA). 3. Confirm no enterprise tenant was provisioned without approval ticket linkage. 4. Measure approval SLA compliance. | Each enterprise provisioning; quarterly compliance review | ServiceNow approval tickets; MSA repository; identity verification documents; SLA compliance report | Solutions Architecture Lead |
| WOS-012 | CC6.3, CC7.2 | **Tenant Inventory Reconciliation** — A comprehensive tenant inventory is maintained and reconciled weekly against all downstream systems (Google Workspace, GCP, Vault, DNS, billing) to detect any discrepancies indicating unauthorized provisioning, orphaned resources, or inventory drift. | Detective | 1. Automated reconciliation job queries each subsystem's tenant list. 2. Results compared against the master Tenant Registry. 3. Discrepancies categorized: orphaned (exists in subsystem but not registry), missing (exists in registry but not subsystem), or mismatched (attribute differences). 4. Discrepancy report sent to Platform Engineering Lead and Security Team. 5. Resolution SLA: Critical (orphaned/missing) within 4 hours; Low (mismatched) within 48 hours. | 1. Review reconciliation reports for the audit period. 2. Verify all discrepancies were resolved within SLA. 3. Deliberately create an orphaned test resource and confirm detection within next reconciliation cycle. 4. Verify report distribution list and escalation path. | Weekly automated; quarterly manual deep reconciliation | Reconciliation job output reports; discrepancy resolution tickets; test resource detection evidence; escalation logs | Platform Engineering Lead |
